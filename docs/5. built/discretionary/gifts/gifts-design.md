# Gifts — Design

> **Purpose:** Captures the approved product direction for the Gifts planner — the problem, the chosen approach, and the key decisions made during requirement refinement. This is the input to `/write-spec`. It is intentionally product-level: no Prisma syntax, no route shapes, no component trees.

## Problem

Households want to plan and track gifts given throughout the year — who they buy for, which events they buy for, how much they intend to spend, and how much they actually spent — and have that planning feed into the waterfall so the gift budget is visible alongside other discretionary spend. Today, gift planning happens in spreadsheets outside finplan, disconnected from the overall plan. The existing `/gifts` page is a "Coming soon" stub, the existing `GiftPerson`/`GiftEvent`/`GiftYearRecord` schema is unused, and the pre-seeded "Gifts" subcategory in Discretionary has no content pipeline.

## Approved Approach

A dedicated Gifts page at `/gifts` organised around the **matrix model** — People × Events, with an allocation at each intersection. The page has three modes selected via vertical tabs in the left panel (below a persistent budget summary block):

- **Gifts** (default) — browse/edit gifts per person via the right-panel drill-down pattern (State 2 = People list → State 3 = Person detail with event cards)
- **Upcoming** — timeline view grouped by month, with non-clickable callout summaries, events showing recipients inline, and a Dateless section at the end
- **Config** — manage Events, People, and household-wide Mode; also the entry point for **Quick Add**, a nested matrix editor for bulk people/events/allocation entry

The planner operates in one of two **household-wide modes**, toggled in Config:

- **Synced** (default) — the Gifts planner owns a single "Gifts" `DiscretionaryItem` that appears in the waterfall via a yearly `ItemAmountPeriod`. The annual budget set in the planner is what flows into the waterfall.
- **Independent** — the Gifts planner runs standalone with no waterfall linkage. Users manage the Gifts discretionary subcategory manually in the waterfall.

The three-layer value model is:

- **Annual Budget** — the user's target for the year; what flows into the waterfall in Synced mode
- **Planned** — sum of all per-gift allocations; can exceed the budget (surfaces an amber over-budget signal)
- **Spent** — sum of recorded actuals; can exceed the budget (surfaces a separate amber over-budget signal)

Rejected alternatives:

- A matrix-first default layout (spreadsheet-style): strong for bulk setup but weak for per-gift detail (dates, notes, status). Matrix lives inside Quick Add instead.
- Keeping gifts entirely outside the waterfall (Independent-only): loses the "gifts are part of my plan" value proposition. Independent remains available as a mode, but Synced is the default.
- Creating one `DiscretionaryItem` per year: rejected in favour of a single item with per-year `ItemAmountPeriod` rows, matching the existing period system.

## Key Decisions

| Decision                                     | Choice                                                                                                                                                                                                                             | Rationale                                                                                                                                                                                             |
| -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Planner ↔ waterfall relationship             | Two household-wide modes: **Synced** (planner owns a single Discretionary "Gifts" item with a per-year `ItemAmountPeriod`) and **Independent** (planner standalone, no waterfall link)                                             | Different households have different preferences; mode is a one-time philosophical choice, not a per-year toggle                                                                                       |
| Mode toggle location                         | Lives in Config as a drill target ("Mode"); not in page header                                                                                                                                                                     | Keeps the main Gifts/Upcoming surfaces calm; editing mode is a deliberate action, not a frequent one                                                                                                  |
| Mode switching                               | Both directions show a confirmation dialog. Synced → Independent deletes the whole `DiscretionaryItem` (including all historical periods); dialog lists what will be destroyed. Independent → Synced creates a fresh item.         | User explicitly chose destructive cleanup over preserving history — "as clean as possible". Clear warning dialog mitigates the destructive nature.                                                    |
| Three-layer value model                      | Annual Budget / Planned / Spent — Budget is what feeds the waterfall (stable); Planned and Spent are planner-side aggregates                                                                                                       | Waterfall shouldn't jitter as allocations drift. "I want to spend up to £X" is the stable anchor; over-allocation is a planning signal, not an accounting fact.                                       |
| Gift states                                  | **Planned / Bought / Skipped**. Entering any spent value (including £0) implicitly marks a gift Bought; clearing the spent field reverts to Planned. Skip via a hover-revealed row action.                                         | Simplest UX for the common "I typed the amount, now it's bought" flow; explicit Skipped state preserves "not this year" records without deleting the planning row                                     |
| "Spent" language exception                   | Anchor #3 and Philosophy Principle 2 updated to allow "spent" **only within the Gifts planner**. The waterfall still never shows "spent".                                                                                          | Gifts is a bounded, discrete, plan-and-reconcile activity. The word "recorded" in place of "spent" would feel unnatural. Scoped narrowly to avoid opening the door to transaction tracking elsewhere. |
| Attention signals                            | Amber dot + text: "planned more than budget by £X" (when Planned > Budget); "spent more than budget by £X" (when Spent > Budget); subtle row-level amber dot when per-gift Spent > Planned                                         | Matches Anchors #10 (non-judgemental), #11 (calm by default), #12 (informational never blocking). No red/green, no "approaching budget" nudge, no waterfall amber bleed.                              |
| Budget staleness check                       | Removed for the gift budget item                                                                                                                                                                                                   | Gift planning is annual by nature; a 12-month staleness check would just add noise to a number that's already expected to be refreshed yearly.                                                        |
| Data shape                                   | **Matrix model** — People and Events are top-level household entities; an allocation exists at each (Person, Event, Year) intersection with `{planned, spent, status, notes, date?}`                                               | Mirrors the spreadsheet users already maintain; cleaner than per-person nested events because adding a new event (e.g. Valentine's) becomes a single column addition across everyone                  |
| Event date categories                        | Two types: **shared-date** (Christmas, Mothers Day — same date every year, stored on the event) and **personal-date** (Birthday, Anniversary — date varies per recipient, stored on the allocation)                                | Matches real-world gift calendars. Allows Upcoming view to resolve dates correctly for both categories.                                                                                               |
| Pre-seeded locked events                     | Birthday (personal), Wedding Anniversary (personal), Valentine's Day (14 Feb), Mother's Day (15 Mar), Easter (10 Apr), Father's Day (15 Jun), Christmas (25 Dec) — all with `isLocked=true`                                        | Covers the common UK gift calendar. Locked prevents accidental deletion/renaming. Users can add custom events alongside.                                                                              |
| Moveable feasts                              | Fixed dates for Easter/Mother's Day/Father's Day (with user override per allocation)                                                                                                                                               | Simplification chosen over implementing lunar/Sunday calculation logic. Listed in Out of Scope for future enhancement.                                                                                |
| Shared gifts (one gift, multiple recipients) | Not supported                                                                                                                                                                                                                      | YAGNI — handled via notes on a single allocation; keeps the matrix model clean                                                                                                                        |
| Household members as gift people             | Auto-surface as `GiftPerson` rows with a `householdMemberId` link and a "Household" badge. If the household member is later removed, the link is nullified but the `GiftPerson` record is preserved.                               | Removes the chore of re-adding household members as gift recipients; preserves gift history when household membership changes                                                                         |
| Year rollover                                | **Automatic on 1 January**, + on-screen notification: "Gift plan for {year} has been created — you may want to review and update the planned amounts." Allocations carry forward (dates, planned amounts); Spent and Status reset. | Gift planning is incremental — "mostly the same as last year, a few changes" is the norm. Notification prompts review without forcing an interrupt.                                                   |
| Previous-year data                           | Preserved as read-only; year selector in page header lets users view historical plans. No editing of prior years.                                                                                                                  | Supports the Synced-mode history invariant (waterfall history must remain intact); lets users compare year-over-year                                                                                  |
| Waterfall gift item structure                | **One** `DiscretionaryItem` ("Gifts") with a new `ItemAmountPeriod` per year (not one item per year)                                                                                                                               | Matches the existing period system for amount changes; avoids proliferating orphaned items                                                                                                            |
| Left panel content                           | Page title + annual budget summary block + three vertical tabs (Gifts · Upcoming · Config). Nothing else.                                                                                                                          | Settings page (`SettingsPage.tsx`) uses the same left-aside mode-switcher pattern — precedent exists. Keeps the left panel calm and focused on navigation.                                            |
| Right panel navigation                       | Right-panel drill pattern (§3.2 of design-system) — State 2 = list, State 3 = detail, with breadcrumb and direction-aware slide transitions. No nested sidebars, no modals.                                                        | Consistent with Income/Committed/Discretionary pages; proven pattern in the codebase                                                                                                                  |
| Quick Add                                    | Nested view drilled in from Config (breadcrumb `← Config / Quick add`), matrix editor with editable cells, scrollable table, row totals, column totals, live summary strip, Save/Cancel                                            | Single place for bulk entry across People/Events/allocations; opens pre-populated so it's a bulk-edit tool as well as bulk-create                                                                     |
| Gift row "needs date" marker                 | Neutral `text-tertiary` label "needs date" after the event name; no amber                                                                                                                                                          | Incomplete-from-quick-entry is a soft state, not an attention signal. Amber reserved for the budget-level nudges.                                                                                     |
| "Planned" vs "Bought" count display          | Planned count uses `text-tertiary` (40%); Bought count uses `text-secondary` (65%). Same cool-white hue — only brightness differs. No new token introduced.                                                                        | Calm, non-judgemental differentiation. Avoids implying "bought" is celebrated. Uses existing tokens — no new design system surface area.                                                              |
| Upcoming view — event grouping               | Shared-date events collapse into a single row with recipients listed inline (e.g. "Christmas: Josh, Mum, Dad +19 more"). Personal-date events stay per-person since dates differ.                                                  | Matches the natural grouping of a real calendar. 26 separate Christmas rows would be noise; one row is scannable.                                                                                     |
| Upcoming callout boxes                       | Non-clickable info cards at the top of the Upcoming timeline: This month / Next 3 months / Rest of year / Dateless. Amounts and counts only, no interaction.                                                                       | User correction — earlier treatment mistakenly made them filters. They are context, not navigation.                                                                                                   |
| Config sub-navigation                        | Drill pattern — State 2 shows three rows (People, Events, Mode), click drills into State 3 for that list. No top tabs.                                                                                                             | Matches the right-panel drill pattern used everywhere else in finplan. No new sub-nav primitive.                                                                                                      |
| Config → People filter label                 | "All / Household / Non-household" (renamed from the proposed "External")                                                                                                                                                           | Friendlier, clearer. "External" implies boundary/security semantics that don't belong here.                                                                                                           |
| Add/edit pattern                             | Inline edit in place (§4.11). Applies to gifts, people, events, mode. No drill deeper from Config → People into a person's gifts.                                                                                                  | Consistent with existing tier pages; Config stays structural, Gifts stays planning.                                                                                                                   |
| New event form — date type choice            | Plain-English radio pair: "Same date every year (e.g. Christmas)" / "Different per person (e.g. Birthday)"                                                                                                                         | Load-bearing choice (wrong selection breaks Upcoming). Radio with examples makes the concept explicit. Implicit "blank = personal" rejected as too magical.                                           |
| Validation                                   | Inline on field blur/change — negative amounts rejected, duplicate names rejected, blank required fields rejected, unsaved-changes confirmation on Cancel. Save-time validation is a safety net.                                   | Better UX — errors surface where they occur, not at submission time.                                                                                                                                  |
| Delete behaviour                             | Cascade delete with confirmation. Locked default events can't be deleted at all.                                                                                                                                                   | Matches existing Committed/Discretionary delete patterns; confirmation prevents surprises.                                                                                                            |

## Out of Scope

- **Auto-calculation of moveable feasts** — Easter, Mother's Day (Mothering Sunday), Father's Day currently use fixed dates. Future enhancement could compute actual UK dates per year.
- **Shared gifts** — one gift for multiple recipients. Handled via notes on a single allocation for now.
- **Gift reminders / calendar notifications** — the Upcoming view surfaces what's next, but no push/email reminders.
- **Gift ideas / wishlist tracking beyond the notes field** — a richer "gift ideas" workflow is not covered.
- **Receipt photos, product links, or purchase tracking** — out of scope; finplan is a planning tool, not a shopping tool.
- **Mobile layout** — inherits the global desktop-first scope (Anchor #6); no dedicated mobile Gifts layout.
- **Import/export of gift plans** — no import from spreadsheet; no gift-plan-specific export.
- **Multi-year gift planning** — only the current year is editable; prior years are read-only; future years don't exist until rollover.
- **Sharing gift plans with other household members** — the Gifts planner is household-scoped, not per-member.

## Design Standard Updates Applied

Two updates were applied to the design standards as part of this design session:

1. **`design-anchors.md` → Anchor #3** — Added the Gifts exception allowing "spent" language within the Gifts planner only. The exception is explicitly scoped to exclude the waterfall and all other tiers/features.
2. **`design-philosophy.md` → Principle 2 (Plan, Not Ledger)** — Mirrored the Gifts exception with a paragraph explaining the reasoning and the narrow scope.

No new design tokens were introduced. "Bought" counts use the existing `text-secondary` token; "Planned" counts use the existing `text-tertiary` token.
