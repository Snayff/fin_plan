What this means mechanically

  • Schema: add DiscretionaryItem.linkedAccountId String? (FK → Account). Keep or remove Account.monthlyContribution? (Open question — see below.)
  • Contribution source of truth: derived at read time — sum(DiscretionaryItem.currentAmount WHERE linkedAccountId = account.id). The Account model stops storing a free-typed number.
  • Where the user sets it: on the waterfall item, not the account. The Assets page shows it read-only with a back-link icon.
  • Growth rate vs user valuation: unchanged. Growth rate drives projections between user-recorded balance snapshots. When the user records a balance, that value becomes the new starting point.
  • Constraint: linking only makes sense for Discretionary items in a "Savings"-type subcategory and Accounts of type Savings / S&S / Pension.
A

Fully derive — remove Account.monthlyContribution

Cleanest model. One source of truth. Requires a migration to drop the field and a back-fill path (no existing users have used it yet, per the placeholder spec).

B

Keep the field — populate from links, allow manual override

Escape hatch for users who don't want to model it in the waterfall. More complex; risks divergence.
