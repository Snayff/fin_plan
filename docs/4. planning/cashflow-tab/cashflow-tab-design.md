# Cashflow Tab — Design

> **Purpose:** Captures the approved product direction — the problem, the chosen approach, and the key decisions made during requirement refinement. This is the input to `/write-spec`. It is intentionally product-level: no Prisma syntax, no route shapes, no component trees.

## Problem

finplan users currently have no way to see how their plan plays out across time. The Overview shows the monthly waterfall (a single snapshot) and the Forecast page shows long-term net-worth projections (decadal horizon), but there is no surface answering the question: **"where are the pinchpoints in my next 12 months?"** — months where yearly bills bunch up, weeks where bills hit before payday, or projected dates where the running balance dips below zero.

A previous yearly-bills cashflow calendar existed (visualising the ÷12 virtual pot model), but it was scoped narrowly to yearly bills, was buried as a right-panel link inside the Overview's Committed section, and was effectively unreachable when the user had no committed yearly bills (the trigger row only rendered when `committed.monthlyAvg12 > 0`). It needs a complete rebuild as a first-class temporal view of the whole plan.

## Approved Approach

Introduce a **Cashflow** view as a sub-section of a restructured **Forecast** page. The Forecast page adopts the two-panel shell (which it currently does not — fixing an existing Anchor 17 deviation): left panel is a section navigator with two entries (Cashflow, Growth), and the right panel renders the selected section full-width.

The **Cashflow** section presents two views:

1. **Year view** — twelve monthly bars showing planned net change per month, anchored to a starting balance read from the user's linked Current/Savings accounts. Bars turn amber when the running balance dips below £0 at any point during that month (overdraft warning). Headline cards show starting balance, projected end-of-window balance, tightest dip, and average monthly surplus.
2. **Month drill-down** — clicking a bar opens a day-level intra-month view: a step-line chart of running balance with dated events (rent, bills, salary, planner purchases) marked, plus a chronological event list with running totals. Discretionary monthly budgets are amortised evenly across the month's days; this is annotated explicitly so the user understands the trace's composition.

The **Growth** section preserves the existing Forecast page's content unchanged (Net Worth, Surplus Accumulation, Retirement) as one stacked right-panel view.

A starting balance is sourced from one or more linked accounts (Savings, or a new "Current" account type). The user picks accounts via a low-prominence button in the Cashflow header — `STARTING BALANCE · £X · N linked accounts ▾`. Replay logic uses the **youngest** linked balance date as the simulation anchor (smallest replay window), and the **oldest** as the staleness signal (worst-case freshness for the user to know about).

This was chosen over the alternatives because:

- A separate top-level "Cashflow" tab would scatter forward-looking surfaces; consolidating under Forecast keeps all temporal views in one place.
- Putting Cashflow in the left panel of Forecast (the user's first instinct) was rejected because the 360px shell width is insufficient for a 12-month chart with day-level drill-down. Section-navigator-on-left, content-on-right honours the shell pattern and gives Cashflow the width it needs.
- A pure surplus-bar view without a starting balance was rejected because the user's stated objective ("bills going out before being paid") fundamentally requires a running-balance trace, which requires a real anchor.

Forward shift unlimited.                                                                                                                                                                                                           | Past months are reviewable through the existing Snapshot feature, not Cashflow.                                                                                                                                                    |
| **Today indicator**                         | Vertical dashed line through the chart at the day-of-month proportion of the first visible bar, with a small `today` label at the top.                                                                                                                                                     | Anchors the user when navigating ranges.                                                                                                                                                                                           |
| **Month drill-down content**                | Step-line chart + event list. Discretionary budgets amortised evenly across the month's days, annotated explicitly via an info chip. Discretionary not listed in the event table — the chip above the chart is sufficient.                                                                 | Step lines accurately reflect "balance after each event"; smooth interpolation would imply continuous change that isn't there. | **Month drill-down navigation**             | Breadcrumb (`← Cashflow / March 2026`) **plus** an inline horizontal month strip (J F M A M J J A S O N D, current month highlighted, tight months in amber).                                                                                                                              | Adjacent-month comparison is a high-value workflow; forcing a breadcrumb round-trip is friction. | **Month drill-down lives in**               | The right panel — replaces the year view via a slide-left transition. Returning to the year view via the breadcrumb is a slide-right transition.                                                                                                                                           | Matches the existing right-panel state machine pattern from design-system §3.2.                                                                                                                                                    |
| **Carry-forward between months**            | **Yes** — each month's projected closing balance becomes the next month's opening balance. The year view's bar colouring uses intra-month balance computed from this carry-forward chain.                                                                                                  | Required to make the running-balance trace meaningful. Necessitated bringing the starting-balance feature into v1 scope.                                                                                                           |
| **Starting balance source**                 | One or more linked Current/Savings accounts. Sum of latest `AccountBalance` for each linked account.                                                                                                                                                                                       | Reuses the existing Account model. Limiting to liquid types prevents pensions/investments inflating the spendable balance.                                                                                                         |
| **Eligible account types**                  | `Savings` and a **new `Current`** enum value. Existing `Pension`, `StocksAndShares`, `Other` are not eligible.                                                                                                                                                                             | Only liquid, day-to-day spendable accounts make sense for cashflow projection. Schema migration adds `Current` to `AccountType`.                                                                                                   |
| **Account selection mechanism**             | Multi-select checklist with a "Select all" master toggle. No explicit "All / Some / Specific" mode — that distinction reduces to "which boxes are ticked".                                                                                                                                 | Simplest mental model. | **Where the linking lives**                 | Inline on the Cashflow page, via a button in the page header. | **Replay anchor date**                      | **Youngest** balance date among linked accounts. Replaying from the oldest date would ignore newer confirmations.                                                                                                |
| **Staleness signal date**                   | **Oldest** balance date among linked accounts.                                                                                                                                                                                                                                             | The projection is partially built on this — that's the worst-case staleness the user should know about.                                                                                                                            |
| **Staleness threshold**                     | Oldest linked balance > 30 days → show banner. | **Staleness banner pattern**                | Matches existing `StaleDataBanner` — thin amber strip with `bg-attention/4 border-b border-attention/8 text-attention`. Text format: `Linked balances X–Y months old · projection may drift · Refresh accounts`. Placement under the title makes the banner feel like part of the Cashflow content area, not a global alert.                                                                          |
| **Pinchpoint definition**                   | Amber bar = the running balance dips below £0 at any point during that month. Direct overdraft-risk warning.                                                                                                                                                                               | Most direct expression of the stated objective. With a real starting balance, "net change < 0" would be a misleading definition (a £4k buffer absorbs negative months without risk).                                               |
| **Discretionary spread treatment**          | Amortised evenly across the month's days (e.g. £933/mo ÷ 31 = ~£30/day). The trace shows the resulting slope. Discretionary is **not** listed in the event table.                                                                   | Dated events alone would understate monthly spend. Front-loading or end-loading would distort the shape. Even amortisation is the most honest single representation, with the chip making the model transparent.                   |
| **Pot model (Anchor 15)**                   | The ÷12 virtual pot model is **preserved** as the underpinning of `committed.monthlyAvg12` in the Overview waterfall. Only the pot's UI visualisation is replaced by the broader cashflow view.                                                                                                                                  |
| **"incl. yearly ÷12" row in Committed**     | **Kept visible** in the Overview waterfall left panel as a passive informational callout. **No longer clickable** — converted from a `<button>` to a non-interactive row.                                                                                                                  | The row is a meaningful piece of context for the Committed total. Removing the click affordance removes the dead entry point without losing the information.                                                                       |
| **Empty / degraded state model**            | Compositional: each missing piece (no linked accounts / no income / no spend) shows a contextual callout near the affected UI element. Multiple callouts compose. A brand-new household sees all three. No separate "first-run" state code.                                                | Render whatever's possible (Anchor 11 / 12 — calm, non-blocking). First-run is just the natural composition.                                                                                                                       |
| **Tab name**                                | Stays `Forecast`. No rename.                                                                                                                                                                                                                                                               | Singular per nav convention; forward-looking semantics fit both Cashflow and Growth.                                                                                                                                               |
| **Data model expansion — day-level timing** | Add a `dueDay` field to yearly bills, monthly committed bills, and a `payDay` field to income sources. These power the intra-month event sequencing.                                                                                                                                       | Required for the "bills before payday" pinchpoint to surface. Additive, non-breaking.                                                                                                                                              |

## Design Standard Decisions

| Standard                                                                                                                      | Resolution                                                                                                                                                                                                                                                                                      |
| ----------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anchor 17** (every non-wizard page uses two-panel shell)                                                                    | Forecast currently violates this. The Cashflow tab feature **fixes the violation** by bringing Forecast into shell compliance.                                                                                                                                                                  |
| **Month strip inside the right panel** (no existing precedent — closest is `SnapshotTimeline` which lives outside the panels) | **Documented exception, scoped to this feature only.** Justification: adjacent-month comparison is a high-value workflow; forcing a breadcrumb round-trip is friction. Design-system docs unchanged. If the same pattern proves useful elsewhere, promote it to a generalisable standard later. |
| **Anchor 1** (planning tool, not ledger) — does linking to real account balances cross the line?                              | No. Reading user-entered `AccountBalance` values is not transaction tracking. The forward projection is still "the plan playing out". The starting balance is a planning anchor, not a live bank position.                                                                                      |
| **Anchor 8** (no bank sync)                                                                                                   | Unaffected. Balances are manually entered into existing `Account`/`AccountBalance` models; no new sync pipeline.                                                                                                                                                                                |
| **Anchor 12** (staleness is informational, never blocking)                                                                    | Honoured. Stale linked balances trigger the amber `StaleDataBanner` strip but never gate the projection.                                                                                                                                                                                        |
| **Anchor 11** (calm by default, amber is the only attention signal, red is for app errors only)                               | Honoured. Overdraft risk uses amber, not red. Stale-data signal uses amber.                                                                                                                                                                                                                     |
| **Anchor 15** (yearly bills use ÷12 virtual pot model)                                                                        | Preserved as a mechanic; only its UI surface is retired.                                                                                                                                                                                                                                        |

## Out of Scope

- **Real bank balance updates / sync** — balances are manually entered via the existing Account flow. No bank integration. (Anchor 8.)
- **Per-account attribution of planned spend** — the plan describes household-level income and spend, not "this comes out of which account". The cashflow projection sums linked balances and treats them as a single buffer.
- **Past-month review on the Cashflow tab** — the year view's earliest position is today. Historical review is the Snapshot feature's job, not Cashflow's.
- **Configurable pinchpoint thresholds** — pinchpoint = balance < £0. No user-configurable cushion (e.g. "warn me if balance drops below £500"). Defer until requested.
- **Notifications / alerts on cashflow events** — no email, no push, no in-app toast for upcoming pinchpoints. The Cashflow page is the only surface; users come to it deliberately.
- **Mobile / sub-1024px viewport support** — desktop-first, per Anchor 6.
- **Modifying the existing Growth section** — Net Worth, Surplus Accumulation, Retirement panels are preserved as-is. They are simply moved from full-width single-column into the right panel of the new shell layout.
- **Snapshot integration** — the Cashflow view is "live plan" only. Viewing a historical snapshot does not change what Cashflow renders. (Snapshot read-only behaviour is per Anchor 16, but the cashflow view is not snapshot-aware in v1.)
- **Settings page entry for linked accounts** — linking is inline on the Cashflow page only.
- **Carry-forward toggle / opt-out** — carry-forward is always on. The £0-baseline mode of earlier discussion was superseded.

## Next Steps (deferred features)

These were discussed and deliberately deferred — not part of this feature, but worth recording as candidates for future iteration:

- **Asset class expansion for cashflow** — allowing investments / pensions to feed cashflow (e.g. drawdown projections in retirement). Currently only liquid Current/Savings accounts are eligible.
- **User-configurable pinchpoint cushion** — e.g. "warn me when balance drops below £500", not just below £0.
- **Cashflow nudges in Overview** — surfacing a "your next month is tight" callout on the Overview page when the projection detects an upcoming pinchpoint. Requires running the cashflow projection on Overview, which adds load; defer until value is proven.
- **Promoting the month strip to a generalisable design pattern** — if the intra-panel temporal strip proves useful elsewhere (e.g. snapshot navigation, quarter views), formalise it in `design-system.md` and remove the documented exception.
- **Snapshot-aware cashflow** — viewing a historical snapshot could replay that snapshot's plan through the cashflow view to show "what was projected at the time of this snapshot".
